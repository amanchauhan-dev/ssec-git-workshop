# Branching in Git

Branching allows working on features independently.

Why use branches:

- Work without affecting main code
- Develop multiple features
- Test safely
- Merge when ready

Main branch usually contains stable code.

---

# Creating a Branch

Create new branch:

```bash
git branch feature-login
```

Switch to branch:

```bash
git switch feature-login
```

Create and switch:

```bash
git switch -c feature-login
```

Used for feature development.

---

# Listing and Deleting Branches

List branches:

```bash
git branch
```

Delete branch:

```bash
git branch -d feature-login
```

Force delete:

```bash
git branch -D feature-login
```

Delete after merging feature.

---

# Merging Branches

Merging combines branches.

Example:

```bash
git switch main

git merge feature-login
```

Changes from feature branch added to main.

Used after completing feature.

---

# Merge Conflicts

Occurs when same lines are edited in two branches.

Conflict example:

```
<<<<<<< HEAD
const tax = 0.18;
=======
const tax = 0.20;
>>>>>>> feature-update
```

Resolve steps:

1. Edit file
2. Keep correct code
3. Save file

Then:

```bash
git add file.js

git commit
```

---

# Fast-Forward Merge

Occurs when no changes exist in main branch.

Git moves pointer forward.

Command:

```bash
git merge feature-branch
```

Creates clean history.

---

# Three-Way Merge

Occurs when branches diverged.

Git creates merge commit.

Command:

```bash
git merge feature-branch
```

Common in team workflows.

---

# Rebase (Advanced)

Rebase rewrites history to make commits linear.

Command:

```bash
git rebase main
```

Moves commits on top of another branch.

Used to maintain clean history.

---

# Interactive Rebase

Allows editing commits.

Command:

```bash
git rebase -i HEAD~3
```

Common actions:

- squash commits
- edit commit messages
- reorder commits

Used before pushing code.

---

# Rebase vs Merge

Merge:

- Keeps full history
- Creates merge commit

Rebase:

- Creates clean history
- Rewrites commits

Rule:

Never rebase shared branches.

---

# Remote Repositories

Remote repository stores code online.

Add remote:

```bash
git remote add origin <url>
```

Check remotes:

```bash
git remote -v
```

Used with GitHub.

---

# Pushing Code to GitHub

Upload changes:

```bash
git push origin main
```

First push:

```bash
git push -u origin main
```

Uploads local commits.

---

# Pulling Changes

Download updates:

```bash
git pull origin main
```

Fetch without merge:

```bash
git fetch origin
```

Keeps local repository updated.

---

# Pull Requests (PR)

Pull Requests allow code review.

Workflow:

1. Create branch
2. Push branch
3. Open Pull Request
4. Review code
5. Merge changes

Used in professional teams.

---

# Git Stash

Temporarily saves changes.

Command:

```bash
git stash
```

Restore stash:

```bash
git stash pop
```

View stash list:

```bash
git stash list
```

Used when switching tasks quickly.

---

# Undo Changes — git restore

Discard changes:

```bash
git restore file.txt
```

Unstage file:

```bash
git restore --staged file.txt
```

Safe way to undo changes.

---

# Undo Commits — git reset

Undo last commit:

```bash
git reset HEAD~1
```

Hard reset:

```bash
git reset --hard HEAD~1
```

Danger:

Hard reset deletes changes.

---

# Safe Undo — git revert

Creates new commit reversing changes.

Command:

```bash
git revert <commit-hash>
```

Used in shared repositories.

---

# git commit --amend

Modify last commit.

Command:

```bash
git commit --amend
```

Used to:

- Fix commit message
- Add missed files

Use before pushing.

---

# Cherry-Pick

Copies specific commit.

Command:

```bash
git cherry-pick <hash>
```

Used when:

- Moving fixes between branches
- Applying selective changes

---

# Git Bisect

Finds buggy commit.

Start:

```bash
git bisect start
```

Mark bad:

```bash
git bisect bad
```

Mark good:

```bash
git bisect good <commit>
```

Uses binary search.

---

# Git Tags

Marks release versions.

Create tag:

```bash
git tag v1.0
```

Push tags:

```bash
git push origin --tags
```

Used for releases.

---

# GitHub Actions (CI/CD)

Automates workflow.

Pipeline example:

Push → Test → Build → Deploy

Used for:

- Running tests
- Building apps
- Deploying projects

Important for modern development.

---

# Best Practices

Follow these rules:

- Write clear commit messages
- Use branches for features
- Never commit secrets
- Pull before pushing
- Keep commits small
- Review code before merging

Professional workflow requires discipline.

---

# Team Workflow

Typical team cycle:

1. Create feature branch
2. Develop feature
3. Commit changes
4. Push branch
5. Open Pull Request
6. Review code
7. Merge branch

Used in real companies.
